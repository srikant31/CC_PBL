## 1. Requirement Analysis (Business Needs)
Requirement Analysis is the process of figuring out what the "Customer" (the user) actually wants the software to do. 
*   **Need 1:** Users want to know if their Python code is "clean" or "messy."
*   **Need 2:** Users want to find "hidden" bugs that might crash their app later.
*   **Need 3:** Users want to find "Security Flaws" so hackers don't steal their data.
*   **Need 4:** Users want a fast, easy-to-use website to get these results.

---

## 2. Requirement Understanding Notes (Technical Implementation)
These are our internal notes on how we actually built those needs into the code:
*   **Understanding 1:** To check for "messy" code, we will use **AST (Abstract Syntax Tree)** to count unused variables and deep loops.
*   **Understanding 2:** To find "hidden" bugs, we will use **Radon** to calculate Cyclomatic Complexity (how many "if/else" branches the code has).
*   **Understanding 3:** To find "Security Flaws," we will use **Bandit** and **OWASP ZAP** to look for common hacker patterns.
*   **Understanding 4:** To make it "Fast and Easy," we will use **Flask** (Backend) and **Bootstrap** (Frontend).

---

## 3. Test Strategy Document (The Big Picture)
The **Test Strategy** is our "Master Plan." We decided that a single test isn't enough. We need **Multiple Layers of Defense**:
1.  **Unit Defense:** Check the math.
2.  **Fuzz Defense:** Attack with "garbage" data to find crashes.
3.  **UI Defense:** Use a robot (Selenium) to click buttons.
4.  **Security Defense:** Act like a "Hacker" using OWASP ZAP.

---

## 4. Test Plan Document (The Logistics)
The **Test Plan** is the "Manager's View" of the project:
*   **Objective:** Ensure the analyzer gives a correct 0-10 score and catches all major security bugs.
*   **Entry Criteria:** All code must be written and "runnable" on a developer's PC.
*   **Exit Criteria:** All 12+ major test cases must pass with **Zero (0) Critical Bugs** remaining.
*   **Tools:** `unittest`, `Hypothesis`, `Selenium`, `OWASP ZAP`, `Bandit`.

## 5. Test Scenarios List (High-Level Ideas)
Test scenarios are "What If?" questions we ask before writing the actual steps:
*   **Scenario 1:** What if the user submits perfect, clean code?
*   **Scenario 2:** What if the user tries to "hack" the system with a script?
*   **Scenario 3:** What if the user submits a huge, 1000-line file?
